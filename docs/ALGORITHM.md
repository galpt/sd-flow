# Algorithm: From scx_flow CPU Scheduler to Diffusion Scheduler

This document explains the cross-domain adaptation of the **scx_flow** CPU scheduling algorithm into the **sd-flow** noise scheduler for Stable Diffusion.

---

## 1. The Original: scx_flow (CPU Scheduling)

[scx_flow](https://github.com/sched-ext/scx/tree/main/scheds/experimental/scx_flow) is a Linux kernel BPF-based CPU scheduler from the sched-ext project. It is intentionally simple and deterministic:

- **No vruntime/vtime** ‚Äî unlike virtually every other scheduler, scx_flow doesn't use weighted fair queuing
- **No heuristics** ‚Äî doesn't reinterpret `nice` values, doesn't detect "latency-critical" tasks
- **No adaptive tuning** ‚Äî no auto-tuning, no load tracking

Instead, it uses a **budget-driven** approach:

### Budget Lifecycle

1. **Sleep accumulation**: While a task is sleeping, it accumulates budget at roughly 1% of sleep time
2. **Budget refill**: On wakeup, `budget += sleep_ns / 100 * weight / 100`
3. **Tier classification**: Tasks are classified into 4 FIFO tiers based on their budget level:
   - `PRIORITY` (budget ‚â• 1500Œºs) ‚Äî long-sleeping interactive tasks
   - `NORMAL` (budget ‚â• 1000Œºs) ‚Äî typical interactive tasks
   - `LOW` (budget ‚â• 500Œºs) ‚Äî short-sleep or partially exhausted
   - `DEFICIT` (budget < 500Œºs) ‚Äî bulk workers, budget exhausted
4. **Rotating dispatch**: A 4-phase rotation cycles which tier is serviced first each round, guaranteeing no tier waits more than 3 dispatch cycles:
   - Phase 0: PRIORITY ‚Üí NORMAL ‚Üí LOW ‚Üí DEFICIT
   - Phase 1: NORMAL ‚Üí LOW ‚Üí DEFICIT ‚Üí PRIORITY
   - Phase 2: LOW ‚Üí DEFICIT ‚Üí PRIORITY ‚Üí NORMAL
   - Phase 3: DEFICIT ‚Üí PRIORITY ‚Üí NORMAL ‚Üí LOW
5. **Slice allocation**: Running tasks get a time slice proportional to their budget (50‚Äì350Œºs)

### Why This Matters

The rotating dispatch provides a **formal starvation guarantee** ‚Äî no tier can be permanently locked out. Combined with deterministic budget accounting, every scheduling decision is fully auditable.

## 2. The Target: Diffusion Noise Scheduling (EDM / Karras)

The [EDM paper](https://github.com/NVlabs/edm) (Karras et al., 2022) defines the standard noise scheduler used in Stable Diffusion UIs:

### Sigma Schedule

The timesteps (noise levels) are generated by:

```
t_i = (œÉ_max^(1/œÅ) + i/(N-1) ¬∑ (œÉ_min^(1/œÅ) ‚àí œÉ_max^(1/œÅ)))^œÅ
```

Where:
- `œÉ_max` = 80.0 (initial noise level)
- `œÉ_min` = 0.002 (final noise level before clean image)
- `œÅ` = 7.0 (controls step spacing ‚Äî higher = more steps near low noise)
- `N` = number of steps (default 18)

When `œÅ = 7`, the schedule heavily concentrates steps in the low-noise region, where the ODE dynamics change fastest.

### ODE Solver

The sampler uses Heun's 2nd order method:

```
1. Euler prediction:  x_next = x + (œÉ_next ‚àí œÉ_cur) ¬∑ (x ‚àí D(x, œÉ_cur)) / œÉ_cur
2. Heun correction:   x_next = x + (œÉ_next ‚àí œÉ_cur) ¬∑ ¬Ω(d_cur + d_prime)
```

Where `D(x, œÉ)` is the denoiser network output.

## 3. The Adaptation: sd-flow

### Core Insight

Both problems share a fundamental structure: **allocating limited compute across multiple competing entities**.

| CPU Scheduling | Diffusion Sampling |
|---|---|
| Tasks compete for CPU time | Timesteps compete for ODE evaluations |
| Budget accumulates during sleep | Budget accumulates across sigma transitions |
| Tier classification by budget level | Phase classification by sigma region |
| Rotating dispatch prevents task starvation | Rotating dispatch prevents sigma-range starvation |

### The Flow Budget for Diffusion

For each sigma transition `œÉ_i ‚Üí œÉ_{i+1}`:

```
ŒîœÉ = œÉ_i ‚àí œÉ_{i+1}
refill = (ŒîœÉ / œÉ_max) √ó (œÉ_i / œÉ_max) √ó 4.0
budget[i+1] = clamp(budget[i] + refill, B_min, B_max)
```

Where:
- `B_max = 2.0` (maximum budget ceiling)
- `B_min = ‚àí0.5` (minimum budget floor, allows slight deficit)
- The `√ó 4.0` scale factor ensures cumulative budget naturally spans `[0, BUDGET_MAX]`, giving meaningful separation between all 4 tiers.

Large sigma drops at high noise levels generate the most budget ‚Äî early denoising steps have the most "creative freedom."

### Tier Classification

Each timestep is classified by its budget:

| Tier | Budget | Sigma Region | Role in Generation |
|---|---|---|---|
| PRIORITY | ‚â• 1.5 | œÉ > 0.75¬∑œÉ_max | Coarse structure, initial burst |
| NORMAL | ‚â• 1.0 | 0.50¬∑œÉ_max < œÉ ‚â§ 0.75¬∑œÉ_max | Medium features |
| LOW | ‚â• 0.5 | 0.25¬∑œÉ_max < œÉ ‚â§ 0.50¬∑œÉ_max | Fine details |
| DEFICIT | < 0.5 | œÉ ‚â§ 0.25¬∑œÉ_max | Final cleanup |

### Rotating Step Allocation

Unlike CPU scheduling (where the rotating dispatch selects which task runs next), diffusion sampling requires sigma values in monotonically decreasing order. The flow scheduler uses rotating dispatch to **distribute step counts across tiers**:

1. Each tier is guaranteed at least 1 step
2. Remaining steps are distributed proportional to the tier's budget share
3. The rotating dispatch ensures the rounding doesn't systematically favor any tier
4. Steps within each tier follow Karras-style polynomial spacing

### Schedule Generation

```
Input: num_steps, sigma_min, sigma_max, rho

1. Segment [sigma_max, sigma_min] into 4 tier sigma ranges
2. Build reference Karras schedule with (num_steps + 1) points
3. Compute budget and tier for each transition
4. Count initial step distribution across tiers
5. Apply rotating dispatch to re-weight distribution
6. Generate Karras-polynomial steps within each tier's sigma range
7. Concatenate in decreasing sigma order, append 0

Output: torch.Tensor [sigma_max, ..., sigma_min, 0]
```

## 4. Key Differences from Karras Schedule

| Property | Karras (œÅ=7) | Flow Schedule |
|---|---|---|
| Step distribution | Concentrated at low noise | Distributed across all sigma tiers |
| Control mechanism | Single parameter `œÅ` | Budget thresholds + rotating dispatch |
| Fairness guarantee | None (œÅ concentrates by design) | Every tier gets at minimum 1 step |
| Determinism | Yes | Yes |
| Heuristics | None | None (budget derived from sigma only) |
